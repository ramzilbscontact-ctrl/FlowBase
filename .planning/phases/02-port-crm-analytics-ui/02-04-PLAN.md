This migration adds a `priority` column to the `tasks` table. The legacy frontend used high/medium/low values. The column was omitted from the original schema (confirmed in 02-RESEARCH.md open questions).

SQL content:
```sql
-- Add priority column to tasks table
-- The original schema omitted this field; the legacy frontend used high/medium/low.
ALTER TABLE public.tasks
  ADD COLUMN IF NOT EXISTS priority text
    NOT NULL
    DEFAULT 'medium'
    CHECK (priority IN ('high', 'medium', 'low'));

COMMENT ON COLUMN public.tasks.priority IS 'Task priority level: high, medium, or low';
```

Use `ADD COLUMN IF NOT EXISTS` so the migration is idempotent (safe to run multiple times).

   ```typescript
   queryKey: ['tasks', filter]
   queryFn: async () => {
     let q = supabase
       .from('tasks')
       .select('id, title, completed, due_date, priority, assigned_to, created_at, profiles!tasks_assigned_to_fkey(full_name)')
       .is('deleted_at', null)

     if (editingTask) {
       // UPDATE path
       const { error } = await supabase.from('tasks').update({
         title: f.title,
         due_date: f.due_date || null,
         priority: f.priority,
         assigned_to: f.assigned_to || null,
       } as any).eq('id', editingTask.id)
       if (error) throw error
     } else {
       // INSERT path
       const { data: { user } } = await supabase.auth.getUser()
       const { error } = await supabase.from('tasks').insert({
         title: f.title,
         due_date: f.due_date || null,
         priority: f.priority,
         assigned_to: f.assigned_to || null,
         owner_id: user!.id,
         completed: false,
       } as any)
       if (error) throw error
     }
   }
   onSuccess: () => {
     qc.invalidateQueries({ queryKey: ['tasks'] })
     toast.success(editingTask ? 'Tâche modifiée' : 'Tâche créée')
     closeModal()
   }
   onError: (e) => toast.error(e.message)
   ```
